# Правила классификации звонков

Документ описывает, как каждый звонок из базы 11 486 получает стадию воронки и причину отвала.

---

## Стадии воронки — кратко

| Стадия | Название | Что произошло |
|---|---|---|
| 0 | Обзвон | Бот совершил звонок |
| 1 | Согласие | Клиент сказал хоть что-то в ответ |
| 2 | Оффер | Бот рассказал о продукте — клиент услышал суть |
| 3 | Встреча | Клиент согласился на встречу с экспертом, назначено время |
| 4 | Квалификация | Клиент сообщил объём своих заявок |

---

## Алгоритм обработки одного звонка

Стадии 0–2 определяются **текстовыми правилами**. Стадии 3–4 и причины — LLM (cached: звонок классифицируется один раз, результат хранится в `classified.json`).

```
Звонок
│
├─ Технический сбой? (no_answer, elevenlabs_hangup, и др.)
│  └─ Да → исключается из воронки, учитывается в Health Monitoring
│
├─ Пустой диалог? (нет текста вообще)
│  └─ Да → Стадия 0, причина не назначается
│
├─ Есть хоть одна реплика user: в диалоге?
│  └─ Нет → Стадия 0 (клиент не ответил)
│
├─ Есть реплика user: → Стадия 1 (Согласие) [текстовое правило]
│
├─ В тексте диалога бот сказал "Это Лариса, б[а/о]тамИн"?
│  (питч-фраза, общая для обоих вариантов скрипта)
│  └─ Да → Стадия 2 (Оффер) [текстовое правило]
│
└─ Применяем LLM-результат из classified.json (если есть)
   │
   ├─ Звонок с питчем (Стадия 2)?
   │  └─ LLM step → max(2, llm_step)  [не понижаем ниже 2]
   │
   ├─ Звонок без питча, LLM определил Стадию 3 или 4?
   │  └─ Применяем LLM step  [бот мог дойти до встречи без питча]
   │
   └─ Звонок без питча, LLM определил Стадию 1 или 2?
      └─ Игнорируем LLM step  [step=2 без питч-текста = LLM-ошибка]
         └─ Причина: LLM (если есть) или текстовые правила
```

**needs_llm (to_classify.json):** все звонки с `user reply` И `duration >= 30s` — ~1 663 записи. Звонки < 30s не могут физически достичь стадии 2+ (питч занимает ~40–60 сек).

**Причины:**
- Стадия 2+ (питч был): из LLM
- Стадия 1 (питча не было): из LLM если есть в classified.json, иначе текстовые правила (confidence 0.6 vs LLM 0.8)
- Стадия 0: причина не назначается

---

## Воронка: 5 стадий

### Стадия 0 — Обзвон
Все звонки в базе. Точка отсчёта воронки.

---

### Стадия 1 — Согласие

Клиент произнёс хоть что-то в ответ — «да», «алло», «нет», «занят» — любое слово.

**Как определяется:** текстовое правило — в диалоге есть хотя бы одна реплика `user:` с непустым текстом.

**Что НЕ считается согласием:** клиент не произнёс ни слова — бот говорил в тишину. Такие звонки остаются на стадии 0.

---

### Стадия 2 — Оффер

Бот полностью донёс суть продукта, и клиент это услышал.

**Как определяется:** текстовое правило — в диалоге бот сказал «Это Лариса, батамИн» (или «ботамИн»). Это питч-фраза, общая для обоих вариантов скрипта. Прощальные фразы говорят «мы батамИн» / «в батамИн» — без «Это Лариса», поэтому не попадают под это правило.

**Вариант А:** «Это Лариса, батамИн. Мы запускаем ИИ-продавца в первую линию продаж...»
**Вариант Б:** «Это Лариса, батамИн. Мм.. у нас команда — люди и ИИ-агенты...»

**Важный нюанс:** если клиент не давал говорить (перебивал, сразу отказал) и бот не успел закончить питч → стадия 1, не 2.

---

### Стадия 3 — Встреча

Назначена конкретная встреча с экспертом.

**Как определяется:** только LLM.

**Признак:** клиент назвал конкретное время И бот подтвердил («пометила», «записала», «отлично, в X часов»).

**Примеры:**
- «Завтра в девять» → стадия 3
- «Пятого июня в тринадцать» → стадия 3
- «Давайте как-нибудь потом» → НЕ стадия 3 (нет конкретики)
- «Перезвоните на следующей неделе» → НЕ стадия 3

---

### Стадия 4 — Квалификация

Бот узнал объём заявок клиента и получил числовой ответ.

**Как определяется:** только LLM.

**Признак:** бот задал вопрос об объёме («сколько заявок в месяц», «сколько лидов», «сколько контактов») И клиент назвал число или диапазон.

**Примеры:**
- «Меньше тысячи» → стадия 4
- «Сколько тысяч?» (встречный вопрос с пониманием смысла) → стадия 4
- «Около пятисот» → стадия 4
- Клиент уклонился, не назвал цифру → стадия 3 (если встреча была)

**Важно:** стадия 4 возможна только если до неё была стадия 3 (встреча). Квалификация без назначенной встречи = стадия 2.

---

## Причины отвала

Причина назначается, когда звонок не дошёл до стадии 4.

**Как определяется:**
- **Стадия 2+** (питч был): причина из LLM — ~323 звонка
- **Стадия 1** (питча не было): причина из LLM если звонок есть в classified.json (duration ≥ 30s), иначе — текстовые правила
- **Стадия 0**: причина не назначается

---

### 1. Молчание
Клиент не отвечал на протяжении всего разговора — только `...` в транскрипте.

Отличие от плохой связи: при молчании бот полноценно говорит, клиент последовательно не реагирует. При плохой связи клиент явно пытается отвечать («Алло? Не слышу вас.»).

---

### 2. Автоответчик / бот
На звонок ответил не живой человек, а AI-секретарь или IVR-меню.

**Признаки:**
- Скриптовые фразы без реакции на содержание: «Говорите, я слушаю.» → «Спасибо. Всего доброго.»
- Повторяющиеся шаблонные ответы вне зависимости от реплики бота
- «В настоящее время [имя] недоступен», «Оставьте сообщение»
- Связные, но явно сгенерированные фразы о погоде, технологиях — не по теме

**Что НЕ является автоответчиком:** живой человек с плохой связью, говорящий «Алло! Не слышу вас» — это реальный человек с проблемой соединения → причина «Занят / неудобно» или «Другое».

---

### 3. Явный отказ
Клиент прямо сказал, что не заинтересован.

**Примеры:** «Не нужно», «Неинтересно», «Не надо», «Мы этим не занимаемся», «Стоп, не продолжать».

---

### 4. Нецелевой клиент
Компания или человек не подходит под профиль: нет отдела продаж, неправильный номер, компания закрылась, человек не принимает такие решения.

**Примеры:** «Вы не туда попали», «У нас нет отдела продаж», «Я не занимаюсь продажами».

Отличие от явного отказа: нецелевой — объективно не подходит. Явный отказ — подходит, но не хочет.

---

### 5. Занят / неудобно
Клиент потенциально заинтересован, но сейчас не может говорить.

**Примеры:** «Я за рулём», «Перезвоните позже», «Сейчас не до этого», «Нужно посоветоваться», «Нужно время подумать».

---

### 6. Против ботов
Клиент отказывается разговаривать с AI и требует живого человека.

**Примеры:** «Соедините с человеком», «Вы живой?», «Не разговариваю с роботами».

Важно: клиент часто является целевым — просто хочет общаться с менеджером. Стоит фиксировать для передачи живому сотруднику.

---

### 7. Уже есть решение
У клиента уже внедрён похожий инструмент или он работает с конкурентом.

**Примеры:** «У нас уже есть своя CRM с автозвонками», «Мы уже работаем с другой компанией по этому направлению».

---

### 8. Агрессия
Клиент реагирует грубо, с нецензурной лексикой или явным раздражением.

---

### 9. Другое
Ни одна из предыдущих категорий не подходит. Аналитик периодически просматривает эту группу для пополнения таксономии.

---

## Текстовые правила для причин стадии 1

Применяются когда питч не был произнесён и LLM-результата нет. Проверяются по репликам клиента в порядке приоритета:

| Порядок | Причина | Ключевые слова в репликах клиента |
|---|---|---|
| 1 | Агрессия | «нахуй», «нахрен», «блядь», «пошёл» |
| 2 | Против ботов | «живой человек», «соедините с человеком», «вы живой» |
| 3 | Автоответчик / бот | «говорите, я слушаю», «оставьте сообщение», «недоступен» |
| 4 | Уже есть решение | «уже есть», «уже работаем», «уже внедрили» |
| 5 | Нецелевой клиент | «не туда», «нет отдела продаж», «не занимаемся продажами» |
| 6 | Занят / неудобно | «занят», «за рулём», «перезвоните», «неудобно» |
| 7 | Явный отказ | «не нужно», «неинтересно», «не надо», «не актуально» |
| 8 | Молчание | все реплики клиента — только `...` |
| 9 | Другое | ничего из выше не совпало |

Уверенность текстовых правил: 0.6 (ниже LLM ~0.8). Записи с текстовыми причинами могут быть уточнены аналитиком.

---

## Как LLM применяет правила

LLM (Claude) читает полный текст диалога и определяет **наивысшую достигнутую стадию** по смыслу, не по ключевым словам. Примеры пограничных случаев:

| Ситуация | Правильная классификация |
|---|---|
| Клиент сказал «Занят» после того как бот произнёс питч | Стадия 2, причина «Занят / неудобно» |
| Клиент много говорил, но бот не дошёл до питча из-за плохой связи | Стадия 1, причина «Другое» |
| Клиент сказал «Окей» на предложение перезвонить с экспертом, но конкретное время не назвал | Стадия 2 (встреча не подтверждена) |
| Клиент ответил на вопрос об объёме без предварительной встречи | Стадия 2 (стадии идут последовательно) |
| Бот сказал «помЕтила» + назвал конкретную дату и время | Стадия 3 |

## Архитектура: тестовое задание vs продакшн

### Как устроено сейчас (тестовое задание)

Все данные обработаны **один раз**, вручную, в рамках одной сессии:

```
XLSX (11 486 звонков за неделю)
    → process.py — текстовые правила, генерирует to_classify.json
    → Claude в сессии — батчи по ~100 диалогов, пишет classified.json
    → process.py снова — мержит LLM-результаты, генерирует data.json
    → React-дашборд читает data.json (статичный файл)
```

`classified.json` — кэш LLM-результатов, пополнялся вручную в ходе работы. Дашборд статичен: чтобы обновить данные, нужно запустить pipeline заново.

### Что нужно для реальной работы

В продакшне данные поступают ежедневно (или в реальном времени), поэтому архитектура должна быть инкрементальной:

**1. Хранилище вместо JSON-файлов**
Postgres или ClickHouse с таблицей `calls`. Каждый звонок пишется один раз, LLM-результат обновляется по `id`. Никакого полного пересчёта.

**2. Очередь классификации**
Новые звонки из CRM/телефонии попадают в очередь (Redis, SQS). Worker читает очередь, прогоняет через LLM (батчами), пишет результат в БД. Дашборд читает из БД — всегда актуален.

**3. Версионирование правил**
Скрипты бота меняются. Нужно хранить версию скрипта и версию классификатора вместе с каждым звонком — чтобы при смене правил не переклассифицировать исторические данные.

**4. Human-in-the-loop в БД**
Исправления аналитика (кнопка «Изменить тег» в дашборде) сохраняются как `override` в отдельной таблице. При агрегации `override` приоритетнее LLM. Накопленные исправления используются для fine-tuning или few-shot примеров.

**5. Мониторинг качества классификации**
Автоматически сэмплировать 1–2% звонков в день для проверки аналитиком. Если доля `Другое` или низкой уверенности растёт — сигнал, что таксономия устарела.
